# 注入人味

> 删壳只是工作的一半。无菌、没声音的写作和机器生成的一样明显——"太稳、太中、太像说明书"本身就是另一种 AI 味。
>
> 这份文件讲的是去味之后，怎么把人味加回来。**仅用于需要作者声音的场景**（公开文/社媒/小说/随笔）。技术文/学术文/文档不需要——中性就是正确的人味。

## 缺灵魂的迹象

去完壳的文本如果还有以下迹象，说明它"技术上干净但没有灵魂"——和机器生成的一样明显：

- 每个句子长度和结构都相同
- 没有观点，只有中立报道
- 不承认不确定性或复杂感受
- 适当时不使用第一人称视角
- 没有幽默、没有锋芒、没有个性
- 读起来像维基百科文章或新闻稿
- 每段都是"观点 + 论证 + 总结"
- 所有情绪都简化成了立场
- 没有跑题、没有题外话、没有半成型的想法

## 怎么注人味

### 有观点

不要只报告事实——对事实做出反应。

❌ `实验产生了有趣的结果。智能体生成了 300 万行代码。一些开发者印象深刻，另一些则持怀疑态度。影响尚不明确。`
✅ `我真的不知道该怎么看待这件事。300 万行代码，在人类大概睡觉的时候生成的。开发社区有一半人疯了，另一半人在解释为什么这不算数。真相可能在无聊的中间某处——但我一直在想那些通宵工作的智能体。`

### 变化节奏

短促有力的句子。然后是需要时间慢慢展开的长句。混合使用。

一个字也可以成句。"对。""不对。""这要看情况。"

长句后面接短句，短句后面接长句。节奏感来自长短交错——不是每句都差不多长。

### 承认复杂性

真实的人有复杂的感受。

❌ `这令人印象深刻。`
✅ `这令人佩服但也有点不安。`

❌ `这是一件好事。`
✅ `我也说不清这是好事还是坏事。`

不把情绪简化为立场——"既佩服又不安"比"佩服"更有人味。

### 适当用第一人称

第一人称不是不专业——是诚实。在公开文/随笔里：

- `我一直觉得……`
- `让我困扰的是……`
- `我拿不准……`
- `说实话，我也没想清楚……`

技术文/学术文不用第一人称——那是场景决定的，不是"不诚实"。

### 允许一些混乱

完美的结构感觉像算法。跑题、题外话和半成型的想法是人性的体现。

- `顺便说一句……`
- `其实我刚才说的不太对……`
- `跑题了，回到正题。`
- `这个观点我还没完全想清楚，但……`

不是每段都需要"观点 + 论证 + 总结"。有些段可以停在半成型的地方。

### 对感受要具体

不是给一个笼统的情绪标签，而是给具体的场景。

❌ `这令人担忧。`
✅ `凌晨三点没人看着的时候，智能体还在不停地运转，这让人不安。`

❌ `这个功能很有用。`
✅ `上周三我用这个功能改了一下午的配置，省了至少三次重启。`

具体场景比抽象判断有人味——"凌晨三点""上周三下午"比"令人担忧""很有用"具体得多。

### 有锋芒

去味不是去棱角。如果作者有态度、有偏见、有锋利——保留。

❌ `这个方案有其优点和不足。`
✅ `这个方案在好的方向上走了一半，然后拐进了死胡同。`

"有其优点和不足"是 AI 式的端水大师。"走了一半拐进死胡同"是有态度的人。

### 不确定的就说不确定

AI 倾向于给确定的结论。人类经常不确定。

- `我不确定。`
- `这事我也没看明白。`
- `可能是 X，但我没把握。`
- `等更多数据吧。`

承认不确定不是软弱——是诚实。

## 注入人味的边界

人味不是表演。以下"人味"是假的，不要加：

### 不要编造经历

不为了像人写的就硬塞个人经历、轶事、童年回忆。如果作者没经历过，不编。`soul.md` 讲的是表达方式上的"人味"，不是内容上的虚构。

### 不要硬塞幽默

不是每篇文章都需要笑话。硬塞的幽默比 AI 味更尴尬。幽默只在作者真有幽默感时才加。

### 不要加感官细节堆砌

"阳光洒在键盘上，咖啡的香气弥漫……"——这种感官细节堆砌是另一种表演，不是人味。具体的细节要有具体的功能（佐证观点/设定场景），不是为了"有细节"而加。

### 不要过度口语化

公开文不是聊天框。"人味"不等于"满篇口语词"。在正式和随意之间找到适合场景的语域——公号文比聊天正式，比论文随意。

## 什么时候不注人味

| 场景 | 注不注人味 |
|---|---|
| 公号文 / 博客 / 随笔 / 专栏 | 注 |
| 小红书 / 微博 / 朋友圈 | 注（但控制力度，别改成散文） |
| 小说 / 故事 | 注（这是声音和节奏的主场） |
| 技术文档 / API 说明 | 不注（中性即正确） |
| 学术论文 / 研究报告 | 不注（客观即正确） |
| 新闻报道 | 不注（中立即正确） |
| 聊天回复 | 最小（直接就行人味，别"注入"） |

## 示范

### 改前（干净但无灵魂）

> 实验产生了有趣的结果。智能体生成了 300 万行代码。一些开发者印象深刻，另一些则持怀疑态度。影响尚不明确。

### 改后（鲜活）

> 我真的不知道该怎么看待这件事。300 万行代码，在人类大概睡觉的时候生成的。开发社区有一半人疯了，另一半人在解释为什么这不算数。真相可能在无聊的中间某处——但我一直在想那些通宵工作的智能体。

**改了什么**：加了第一人称观点（"我真的不知道"）、承认了复杂性（"一半疯了一半在解释"）、给了具体场景（"人类大概睡觉的时候"）、收在了未解决的张力上（"但我一直在想"）而不是确定结论。

### 改前（干净但无灵魂）

> 这个功能有其优点和不足。它提高了效率，但也增加了复杂度。用户需要根据自身需求权衡。

### 改后（有锋芒）

> 这个功能在好的方向上走了一半，然后拐进了死胡同。效率确实上去了，但复杂度也跟着翻了一倍——值不值，看你能不能接住这个复杂度。

**改了什么**：把端水大师的"优点和不足"换成了有态度的判断（"走了一半拐进死胡同"）、给了具体权衡标准（"看你能不能接住这个复杂度"）而不是泛泛的"根据需求权衡"。
